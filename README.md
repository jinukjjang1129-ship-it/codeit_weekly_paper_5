# Weekly Paper: 데이터베이스 정규화 및 논리적 모델링

## 1. 데이터베이스 정규화란?

정규화는 관계형 데이터베이스 설계에서 데이터 중복을 최소화하고 무결성을 유지하기 위해 데이터를 구조화하는 프로세스다. 핵심은 하나의 테이블에는 하나의 주제만 담는 것이며, 이를 위해 큰 테이블을 여러 개의 작은 테이블로 분할해 나가는 과정을 거친다.



### 정규화의 목적
- 이상 현상 제거: 삽입, 삭제, 갱신 시 발생하는 논리적 오류를 방지한다.
- 데이터 구조의 안정성: 새로운 데이터 형식이 추가되어도 구조 변경을 최소화할 수 있다.
- 저장 공간의 효율화: 중복된 데이터를 제거하여 용량을 아낀다.

---

## 2. 정규화의 장단점

### 장점
- 데이터 무결성 유지: 중복 데이터가 없어 수정 시 데이터 불일치가 발생하지 않는다.
- 이상 현상 방지: 삽입/삭제 시 원치 않는 정보가 사라지거나 누락되는 문제를 해결한다.
- 유연한 구조: 데이터베이스 확장이 용이하고 구조가 단순해진다.

### 단점
- 조인 비용 증가: 테이블이 분리되므로 데이터를 합쳐서 볼 때 JOIN 연산이 많아져 성능이 저하될 수 있다.
- 쿼리 복잡도: 데이터를 조회하기 위한 SQL 문장이 길고 복잡해질 수 있다.

---

## 3. 논리적 모델링의 주요 구성 요소

비즈니스 요구사항을 데이터베이스 구조로 추상화하는 단계에서 사용되는 핵심 요소다.



### 엔터티 (Entity)
데이터로 관리하고자 하는 실체나 객체를 의미한다.
- 예시: USERS(사용자), ARTISTS(아티스트), VOTES(투표)

### 속성 (Attribute)
엔터티가 가지는 최소 단위의 정보로, 파일 시스템의 필드에 해당한다.
- 예시: USERS 엔터티의 속성 -> user_id, email, credits

### 관계 (Relationship)
엔터티 간의 논리적인 연결 고리를 의미한다.
- 예시: 한 명의 사용자가 여러 번 투표를 할 수 있는 1:N 관계

---

## 4. 예시 쿼리

정규화된 구조에서 테이블을 생성하고 관계를 활용해 데이터를 조회하는 기초 SQL이다.

### 테이블 생성 (엔터티 및 관계 정의)
CREATE TABLE users (
    user_id INT PRIMARY KEY,
    email VARCHAR(100),
    credits INT DEFAULT 0
);

CREATE TABLE votes (
    vote_id INT PRIMARY KEY,
    user_id INT,
    artist_id INT,
    vote_count INT,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
);

### 데이터 조회 (JOIN 활용)
-- 특정 사용자의 투표 현황과 남은 크레딧 조회
SELECT u.email, u.credits, v.artist_id, v.vote_count
FROM users u
JOIN votes v ON u.user_id = v.user_id
WHERE u.user_id = 1;

## 요약
데이터베이스 설계 시 정규화는 데이터의 정확성을 위해 필수적인 과정이다. 하지만 서비스의 규모와 조회 성능을 고려하여 적절한 수준의 정규화를 진행하는 것이 중요하다. 엔터티, 속성, 관계를 명확히 정의하는 논리적 모델링이 선행될 때 견고한 데이터베이스를 구축할 수 있다.
